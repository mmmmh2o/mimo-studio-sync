### 🤖 AI

Agent-king 👑



一句话概括：你给它一个想法，它自动拆分成子任务，分配给多个 AI Worker 并行开发。



核心流程


text
text
你的想法 → Splitter (LLM拆分) → tasks.json
                                    ↓
       Scheduler (5s循环调度) ← 读取任务
            ↓
       分配给空闲 Worker (tmux隔离)
            ↓
       Monitor (30s轮询) → 异常告警
            ↓
       Hooks → WebSocket → Dashboard 实时展示



架构模块


层	模块	职责
核心	splitter.ts	LLM 把大想法拆成子任务

	scheduler.ts	5s 循环，按优先级分配任务给 Worker

	worker.ts	每个 Worker 一个 tmux session + git worktree 隔离

	monitor.ts	30s 轮询检测超时/空闲/错误
库	llm.ts	LLM API 调用封装

	tmux.ts	tmux session 管理

	worktree.ts	Git worktree 隔离

	store.ts	JSON 存储 + JSONL 追加

	alert.ts	三级告警回调
展示	dashboard.ts	深色主题 Web 面板

	server.ts	HTTP + WebSocket 实时推送


用法


bash
bash
agent-king run "做一个能自动记账的 Telegram bot" --workers 3
# → 自动拆分 → 3个Worker并行开发 → Dashboard: localhost:3456



技术栈


TypeScript / Node.js >= 20
tmux 隔离 Worker、git worktree 隔离代码
WebSocket 实时推送事件
依赖 OpenAI API（拆分）+ Claude Code（执行）

🔵 src/index.ts — 入口 & CLI



职责：命令行入口，解析参数，协调各模块。



命令：run（一句话启动）、resume（断点继续）、split（只拆分不执行）、status、report、serve、config、stop
参数解析：--workers、--dir、--model、--port、--no-web
run 流程：拆分任务 → 打印任务列表 → 等用户确认（Enter）→ 启动 server + reporter + checkpoint → 启动 scheduler
resume 流程：加载 checkpoint → doing→todo 重置 → 启动 scheduler




🔵 src/lib/coder.ts — AI 代码生成器（核心新增）



职责：调 LLM 生成代码，写入文件，自动 git commit。




关键设计：

System Prompt 要求 LLM 返回严格 JSON：{files: {"path": "content"}, message, done}
read_project_context() — 扫描项目已有代码文件（限 20 个、单文件 <10KB），喂给 LLM 做上下文
parse_coder_response() — 处理 markdown 代码块包裹等 LLM 输出异常
write_files() — 路径穿越保护：拒绝 ..、绝对路径，确保写入在项目目录内
auto_commit() — 每次写完自动 git add -A && commit，author 是 agent-king
execute_task() — 主入口：读上下文 → 构造 prompt → 调 LLM → 解析 JSON → 写文件 → commit




🔵 src/lib/config.ts — 配置系统



职责：JSON 配置文件替代环境变量。




查找优先级：项目/agent-king.json → 项目/.agent-king.json → 当前目录 → ~/.agent-king/config.json → 环境变量 → 默认值




配置结构：

json
json
{
  "llm": { "api_key": "...", "base_url": "...", "model": "gpt-4o-mini" },
  "worker": { "model": "mimo", "count": 3 },
  "server": { "port": 3456 }
}




带缓存、reset_config() 测试用。





🔵 src/lib/llm.ts — LLM API 调用



职责：统一的 LLM 调用接口。



从 config.ts 读取 API Key / base_url / model
⚠️ 用 execSync(curl) 调 API — 同步阻塞，120s 超时
支持 OpenAI 兼容端点（OpenAI / DeepSeek / MiMo 等）
返回 {content, usage}



注意：这里回退到了 execSync(curl) 而非之前修的 fetch() async，可能是重构时覆盖了。





🔵 src/core/splitter.ts — 任务拆分



职责：用户一句话想法 → LLM → 子任务列表。



System Prompt 要求返回 [{title, description, priority, dependencies}]
自动分配 ID：task-001、task-002...
写入 tasks.json
提供 load_tasks() 和 update_task() 供其他模块读写




🔵 src/core/scheduler.ts — 调度器



职责：5s 循环调度，分配任务给空闲 Worker。



启动：初始化 workers → 为每个 worker 创建 git worktree → 启动 monitor → 5s 定时调度
调度逻辑：

1.
检查是否全部完成（全部 done/error/skipped → 打印摘要 → 停止）
2.
找空闲 worker → 按优先级挑 todo 任务 → 检查依赖 → 分配
3.
WebSocket 推送进度
依赖检查：dependencies 中所有任务都 done 才可执行
停止：清理 worktree → 保存 checkpoint




🔵 src/core/worker.ts — Worker 管理



职责：Worker 状态管理 + 异步任务执行。




关键变化（重构后）：

不再依赖 tmux — 不再创建 tmux session
用 Promise 跟踪执行 — running_tasks: Map<worker_id, Promise>
assign_task()：更新状态 → 异步调 coder.execute_task() → Promise resolve 后 complete_task() 或 fail_task()
自动重试：失败任务 retry_count < max_retries 时重置为 todo（最多 3 次）
check_worker_completion()：检查 Promise Map，不再检查 tmux session




🔵 src/core/monitor.ts — 监控器



职责：30s 轮询检测异常。



任务超时：>30min 标记 error，可自动重试
心跳更新：每轮更新 worker 的 last_heartbeat
进度快照：写入 progress.json（完成数/进行中/失败/预估剩余时间）
预估剩余：基于已完成任务的平均耗时




🔵 src/lib/store.ts — 数据存储



职责：简单的 JSON 文件 + JSONL 追加写。



read_json() / write_json() — 读写 JSON 文件（带 fallback）
append_jsonl() — 追加写 JSONL（事件日志）
read_jsonl() — 读取全部 JSONL 行




🔵 src/lib/alert.ts — 告警系统



职责：三级告警 + 回调分发。



alert_error / alert_warn / alert_info
注册 handler（server 用它推 WebSocket 告警横幅）
终端输出带 emoji 前缀（🔴 🟡 🟢）




🔵 src/lib/reporter.ts — 进度报告



职责：每 30s 终端打印进度。



进度条 [████████░░] 40%
显示进行中任务 + 耗时、最近完成、异常任务
格式化时间（s / m / h）




🔵 src/lib/checkpoint.ts — 断点保存/恢复



职责：60s 自动保存 + 从断点恢复。



保存：idea + tasks + workers + 统计 → checkpoint.json（旧文件备份为 .bak）
恢复：doing→todo 重置（因为 Worker 已不存在）
has_checkpoint()：有未完成任务才算有效




🔵 src/lib/worktree.ts — Git Worktree 隔离



职责：每个 Worker 一个独立 worktree 分支。



create_worktree()：自动 init git → 创建 agent-king/worker-N 分支 → worktree add
cleanup_all()：停止时清理所有 worktree
分支已存在时有 fallback 逻辑




🔵 src/lib/tmux.ts — tmux 封装（遗留）



职责：tmux session CRUD。重构后不再被核心模块引用，但代码保留。



new_session / send_keys / capture_pane / kill_session / list_sessions




🔵 src/server.ts — HTTP + WebSocket 服务器



职责：Web API + 实时推送。



API 路由：

GET /api/tasks / GET /api/workers / GET /api/progress / GET /api/events
POST /api/hooks — 外部事件回调
POST /api/tasks/:id/retry — 重试失败任务
DELETE /api/tasks/:id — 跳过任务
GET / — Dashboard HTML
WebSocket：/ws 路径，推送 init/progress/event/alert/task_update




🔵 src/dashboard.ts — Dashboard HTML



职责：内联的完整 Web Dashboard（~13KB 单个模板字符串）。



GitHub 风格深色主题
4 个统计卡片（完成/进行中/待执行/失败）
任务表格（含重试/跳过按钮）
环形进度 + 进度条
Worker 状态列表
实时事件流 + 最近活动时间线
WebSocket 自动重连 + 30s 备用轮询





总结一句话：重构后的架构更简单了 — 去掉 tmux，Worker 直接用 Promise + LLM API 写代码，但 llm.ts 回退到了 execSync(curl) 同步调用，这是个需要关注的点。
