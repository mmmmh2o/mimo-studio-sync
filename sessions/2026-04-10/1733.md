多轮对话 + 变量在这个场景下的运作方式


text
text
变量池（数据库 + 运行时）:
{
  "project_name": "订单管理系统",
  "requirements": [从网页抓取的需求列表],
  "tech_stack": "Python + FastAPI + SQLite",
  "current_step": 3,
  "prev_code": "上一轮生成的代码...",
  "error_msg": "上一轮的报错...",
  "test_result": "PASS/FAIL"
}

第1轮 → 发送: "项目: {{project_name}}，需求: {{requirements}}，技术栈: {{tech_stack}}"
        → 收到: 系统设计方案
        → 提取: design_doc → 变量池

第2轮 → 发送: "根据以下设计 {{design_doc}}，生成数据库表结构和API接口定义"
        → 收到: SQL + API spec
        → 提取: schema → 变量池, api_spec → 变量池

第3轮 → 发送: "根据 {{schema}} 和 {{api_spec}}，实现以下核心模块的代码：..."
        → 收到: 代码文件
        → 保存: 写入项目目录

第4轮 → 发送: "为以下代码写单元测试：{{prev_code}}"
        → 收到: 测试代码
        → 执行: 跑测试 → {{test_result}}

第5轮 (条件分支):
  if test_result == FAIL:
    → 发送: "测试失败，错误: {{error_msg}}，请修复代码"
    → 循环回到第4轮
  else:
    → 结束，通知你

技术选型



做成一个桌面应用，本质就是带自动化能力的定制浏览器：



方案	原理	优劣
Electron	Chromium + Node.js，puppeteer 内置	成熟稳定，包体积大 (~150MB)，JS 生态
Tauri	系统 WebView + Rust 后端	极小 (~10MB)，WebView 兼容性差些
CEF (Python)	嵌入 Chromium，Python 控制	Python 友好，但 CEF 绑定不太成熟
Python + Playwright	打开系统 Chrome，Python 做侧边栏 overlay	最轻量，但 UI 要额外搭



我推荐 Electron，理由：



1.
就是 Chromium，MiMo 不会有任何兼容问题
2.
puppeteer-core 直接控制自己的浏览器内核，零外部依赖
3.
侧边栏 UI 就是 HTML/CSS/JS，随便做
4.
人工接管天然支持——同一页面，暂停脚本就行
5.
跨平台（Linux/Mac/Win）



或者 Tauri，如果你在意包体积和性能。

改进 3：变量类型系统


text
text
现在: 变量全是字符串，没有类型

改进后:

变量类型:
├── string      "hello"
├── number      42
├── boolean     true
├── array       ["a", "b", "c"]
├── object      { "key": "value" }
├── file        /path/to/file

变量转换器:
├── {{reply}} → string
├── {{reply | extract_code}} → code (提取代码块)
├── {{reply | extract_json}} → object (提取 JSON)
├── {{reply | extract_table}} → array (提取表格)
├── {{file_content | split_lines}} → array
└── {{data | jsonpath:$.items[*].name}} → array
