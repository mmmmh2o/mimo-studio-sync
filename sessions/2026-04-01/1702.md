文件格式对照


软件	文件格式	说明
GX Developer	.gpp (工程) / .gxw (工作区)	老格式
GX Works2	.gxw2	独有格式
GX Works3	.gx3	独有格式
GX IEC Developer	.gxg	IEC标准项目


兼容性


操作	是否支持
GX Developer → GX Works2	✅ 可以导入，File → Open Other Format File
GX Works2 → GX Developer	❌ 不行，单向升级
GX Works2 → GX Works3	✅ 可以导入（仅Q/L系列程序，FX有限制）
GX Works3 → GX Works2	❌ 不行
GX Developer → GX Works3	✅ 可以导入
不同GX Works2版本之间	⚠️ 高版本工程低版本打不开，保存时可选"另存为低版本"但会丢功能


实际要注意的坑


1.
向下兼容是单向的 — 高版本能开低版本，反过来不行
2.
另存为低版本 — GX Works2里可以 File → Save As → 选更早版本格式，但部分新功能块会丢失或报错
3.
不同CPU系列不能混 — FX工程和Q工程在GX Works2里是分开的模板，不能直接改CPU类型就换
4.
库文件 — .gxl (库文件) 也是版本绑定的，高版本库低版本用不了


建议


现场维护老设备 → GX Developer（老Q/FX系列还能用）
FX系列新项目 → GX Works2 即可
Q系列新项目 → GX Works2 或升级到 GX Works3
iQ-R/iQ-F系列 → 必须用 GX Works3

### 🤖 AI

方案A：逆向生成二进制.wpg



优点：

生成的文件能直接用GX Developer打开，体验最接近"原生"
不依赖任何外部工具，纯Python搞定
生成速度快，适合批量生成程序



缺点：

二进制格式没公开文档，我是从你这一个文件逆向推断的
只覆盖了基础指令（LD/AND/OR/OUT/SET/RST/TMR/CNT），功能指令（MOV/CMP/ZCP等）还没分析
不同GX Developer版本可能有格式差异，兼容性无法保证
没法处理注释、SFC、FB等高级功能
出了问题很难调试——你不知道是格式错误还是逻辑错误



适合： 简单逻辑程序、批量生成模板、快速验证想法





方案B：指令表文本 + 工程模板



优点：

指令表（IL/STL）格式是三菱公开标准，完全可靠
文本可读、可版本管理（git）、可diff
指令集全覆盖，包括所有功能指令、应用指令
注释、声明都能带上
生成的文本可以手动粘贴到GX Developer指令列表编辑器



缺点：

不是"直接双击打开工程"的体验，需要手动导入
工程模板（Gppw.gpj等）里的参数（CPU型号、通讯设置等）还是需要手动配
没法直接预览梯形图效果，要导入后才能看



适合： 正经项目开发、需要可靠性和可维护性





方案C：生成GX Works2格式（.gxw2）



优点：

GX Works2是当前主流，覆盖FX/Q/iQ-R系列
.gxw2本质是zip包里的XML，结构清晰，容易生成
支持注释、变量表、FB、多程序块等现代功能
生成的文件直接GX Works2打开就是完整工程
GX Works2可以另存为兼容老版GX Developer



缺点：

XML结构也比较复杂，需要完整分析GX Works2工程结构
同样没有官方文档，需要逆向分析
你的老文件是GX Developer格式，如果目标设备是老FX系列，用GX Works2可能有型号限制
你本地可能没有GX Works2（只有GX Developer）



适合： 新项目、需要完整工程体验、设备支持GX Works2





我的建议



先走B，再拓展到A/C。




理由：

1.
指令表格式是标准的，不存在兼容性问题，0风险

3.
等指令表生成稳定了，再用它作为"中间格式"去生成A或C的二进制文件
4.
实际工程里，指令表文本也方便做代码审查和版本管理
